Don’t repeat yourself

Speaker: Erika Maer

Authors

The principle has been formulated by Andy Hunt and Dave Thomas in their book The Pragmatic Programmer.

Andy Hunt

Dave Thomas

DRY & SPR

“A class should have only one reason to change”

SSOT - single source of truth

In information systems design and theory, SSOT is the practice of structuring information models and associated data schema such that every data element is mastered (or edited) in only one place.

Modular architecture

  • Draw a graphical diagram of your system, divide it into visual components. Complex projects may require such a hierarchy per component.
  • Before writing the code, mark it in your graphical diagram, determine its role among other components of the system.
  • Be clear about the views that your code will give out to other parts of the system, and which it should hide (private / public).
  • Make sure your code is loosely coupled to other views of the system - being tightly coupled is very bad for the overall architecture.

When the principle doesn't work

DRY & WET

Write Everything Twice (WET) is a cheeky abbreviation to mean the opposite i.e. code that doesn’t adhere to DRY principle.

Abuse of the principle

The implicit drawback is that if you are DRY-aware, you might accidentally combine similar logic rather than duplicating the implementation.

Examples

Selector grouping

Bad practice

              
                h1 {
                    color: #94b5ff;
                    font-size: 20px;
                }
                p {
                    color: #94b5ff;
                    font-size: 20px;
                }
                .title {
                    color: #94b5ff;
                    font-size: 20px;
                }
                h4 {
                    color: #94b5ff;
                    font-size: 20px;
                }

Good practice

              
                    h1, p, .title, h4 {
                        color: #94b5ff;
                        font-size: 20px;
                    }

Cycles

Bad practice

              
              array[0].toUpperCase();
              array[1].toUpperCase();
              array[2].toUpperCase();
              array[3].toUpperCase();

Good practice

              
              for (let i = 0; i < array.length; i++) {
                  array[i].toUpperCase();
              }

Functions

Bad practice

              
              const time = new Date();
              const hour = time.getHours()

              if (hour > 6 && hour < 11) {
                  console.log('Good', 'morning!', 'Now:', hour, 'hours');
              } else {
                  console.log('Good', 'evening!', 'Now:', hour, 'hours');
              }

Good practice

              
                const time = new Date();
                const hour = time.getHours()

                if (hour > 6 && hour < 11) {
                    greeting('morning!');
                } else {
                    greeting('evening!');
                }

                function greeting(dayTime) {
                    console.log('Good', dayTime, 'Now:', hour, 'hours');
                }
